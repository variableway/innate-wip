# Innate 定位 V2: 个人技术创作平台

## 核心定位修正

不是"工具展示"，而是**"个人技术创作过程的可视化平台"**

```
WaytoAGI: 用别人的工具    →  工具导航
BestBlogs: 读别人的文章   →  内容筛选  
Innate:    学我的过程，用我的创造  →  个人技术品牌
```

## 三层内容架构

### Layer 1: Feeds（输入+见解）
不是简单的链接聚合，而是：
```
文章链接
+ 我的关键摘录
+ 我的思考/质疑
+ 和我已有知识的关联
= 有观点的阅读笔记
```

**示例**：
```markdown
## 读了：Claude Code 架构解析
[原文链接]

**关键洞察**：
- 本地文件管理是AI编程的核心
- 上下文治理比Prompt更重要

**我的想法**：
这和我在 spark-cli 中做的批量Git管理有共通之处——
AI也需要"结构化"的上下文才能有效工作。

**关联**：
→ 可以改进我的文件整理工具，加入"AI可理解"的索引
```

### Layer 2: Tutorials（可运行的创造）
不是"教你写代码"，而是：
```
我的工具代码
+ 一键运行方式（本地/云端）
+ 修改指南
+ 灵感来源（和Feeds的关联）
= 可复制的创造过程
```

**运行方式**：
- **本地**：`git clone && ./run.sh`
- **云端**：一键部署到 Vercel/Render
- **Web**：直接浏览器运行（WebContainer）

### Layer 3: Process（思考透明）
展示"从输入到输出"的完整链条：
```
Week 12
├── 输入：读了X篇东西
├── 思考：产生Y个想法  
├── 实践：做了Z个工具
└── 复盘：什么有效，什么无效
```

## 和纯技术博客的区别

| 维度 | 技术博客 | Innate |
|------|---------|--------|
| **内容** | 文章 | 文章 + 可运行代码 |
| **交互** | 阅读 | 阅读 + 运行 + 修改 |
| **关系** | 作者→读者 | 创作者→同路人 |
| **目标** | 传播知识 | 展示过程，邀请参与 |

## 长期愿景：从个人到社区

### 阶段1：个人品牌（现在）
- 我一个人输出
- 建立"可复制、可运行"的标准
- 积累内容模板和工作流

### 阶段2：邀请制（6-12个月后）
- 邀请其他人用Innate格式分享
- 保持质量：必须通过"5分钟可运行"审核
- 我作为策展人

### 阶段3：平台化（1-2年后）
- 任何人可以创建自己的Innate页面
- 但核心保持：Feeds + Tutorials + Process
- 可能的开源/自托管选项

## 关键UX原则

### 1. 即时性原则
每个Tutorial必须能：
- **30秒内**看到效果预览
- **2分钟内**跑起来
- **5分钟内**理解并能修改

### 2. 过程透明原则
每个创造必须展示：
- 灵感来源（哪篇Feeds）
- 迭代过程（v1→v2→v3）
- 失败尝试（什么没成功）

### 3. 关联性原则
内容之间必须有关联：
- Feeds ↔ Tutorials（读了什么→做了什么）
- Tutorials ↔ Tutorials（工具A是工具B的基础）
- Process ↔ 时间线（周与周之间的演进）

## 重新评估UI建议

### ✅ 保留的建议

**双栏Feed布局**
```
左侧：周/时间线导航
右侧：本周的输入→输出完整流程
```
理由：展示"过程"，不是列表

**教程的可运行性**
```
每个教程顶部必须有：
[🚀 在线运行] [💻 本地运行] [📋 复制代码]
```
理由：核心差异化

**首页定位说明**
```
"记录我的技术探索，分享可立即运行的创造"
```

### ❌ 去掉的建议

- 可折叠Sidebar：分散注意力
- 全局搜索：内容<50时用浏览更高效
- 复杂的分类系统：扁平化，用标签

### 🔄 新增的建议

**Feeds文章的个人化展示**
```
不是：
- 标题
- 摘要
- 链接

而是：
- 原文标题
- 我的关键洞察（高亮）
- 我的批注/质疑
- 关联到我做的工具
```

**Tutorial的运行状态指示**
```
[✅ 已验证可运行 - 2024-03-27]
[⚠️ 需要更新 - 依赖版本过期]
```

**Process时间线的可视化**
```
2024 Week 12
├── 📥 输入 (3篇文章)
├── 💡 洞察 (2个想法)
├── 🔧 输出 (2个工具)
└── 📊 数据 (节省时间: 30min/天)
```

## 内容创作SOP

### 每周流程

**周一**：整理上周的Feeds
- 收集阅读过的文章/视频
- 提取关键洞察
- 写下我的想法

**周三**：发布一个小工具
- 基于Feeds的灵感
- 确保5分钟可运行
- 写好Tutorial

**周五**：Process复盘
- 输入→输出的完整回顾
- 什么有效，什么无效
- 下周计划

## 技术架构调整

### 新增：可运行环境检测
```typescript
// 检查教程是否可在线运行
interface RunnableCheck {
  local: boolean      // 有 run.sh / Makefile
  cloud: boolean      // 有部署链接
  web: boolean        // 纯前端可在浏览器运行
  lastVerified: Date  // 最后验证时间
}
```

### 新增：Feeds-Tutorial关联
```typescript
interface Feed {

  title: string
  url: string
  myInsights: string[]    // 我的洞察
  myQuestions: string[]   // 我的疑问
  relatedTutorials: string[] // 关联的工具
}
```

## 总结：Innate 是什么

> **一个人的技术探索周记，但每一篇都可以被复制、运行、修改。**

不是博客，因为博客只能读。
不是文档，因为文档只讲结果。
不是工具站，因为工具站没有过程。

Innate 是：**过程透明 + 结果可运行 + 邀请参与**。

"这就是我本周读的东西，产生的想法，做的东西——你也可以这么做。"
